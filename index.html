§ 01

Java Evolution Context

🔥 The Problem Java 8 Solved

Before Java 8, writing concurrent, expressive, collection-processing code required verbose boilerplate. Anonymous inner classes, external iteration, null handling — all were pain points. Java 8 introduced a functional programming model alongside the OOP paradigm, not replacing it.

The driving force: multicore processors became the norm. Java needed a way to make parallelism easy without manual thread management. Streams + Fork/Join pool solved exactly this.

JAVA 1.0–4 (1996–2002)
OOP Foundation
Classes, interfaces, garbage collection. Single-paradigm: pure object-oriented.
JAVA 5 (2004)
Generics, Enums, Annotations
Type safety added. for-each loop introduced. Foundation for Stream API later.
JAVA 7 (2011)
Fork/Join, try-with-resources
Fork/Join framework planted the seed for parallel streams. Diamond operator added.
JAVA 8 (2014) ← YOU ARE HERE
Functional Revolution
Lambdas, Streams, Optional, new Date/Time, default methods, CompletableFuture. The biggest Java release since generics.
JAVA 9–21 (2017–2023)
Builds on Java 8
Records, sealed classes, pattern matching, virtual threads — all standing on the functional foundation Java 8 laid.

Before vs After Java 8

JAVA 7 — Filter + Sort (Verbose)
// Java 7: External iteration, verbose, error-prone
List<String> names = Arrays.asList("Alice", "Bob", "Charlie", "Dave");
List<String> result = new ArrayList<>();

for (String name : names) {
    if (name.length() > 3) {
        result.add(name.toUpperCase());
    }
}
Collections.sort(result, new Comparator<String>() {
    @Override
    public int compare(String a, String b) {
        return a.compareTo(b);
    }
});
JAVA 8 — Same Logic (Expressive)
// Java 8: Internal iteration, declarative, readable
List<String> result = names.stream()
    .filter(name -> name.length() > 3)
    .map(String::toUpperCase)
    .sorted()
    .collect(Collectors.toList());
ℹ️ Key Paradigm Shift
Java 8 moved from HOW (imperative: explicit loops, mutations) to WHAT (declarative: describe the desired result). This is the essence of functional programming layered onto Java's OOP model.

Lambda Expressions

📌 Definition

A lambda expression is an anonymous function — a concise representation of a single-method interface (functional interface) that can be passed as a value, stored in a variable, or returned from a method.

Internal Mechanism

Lambdas are NOT syntactic sugar for anonymous inner classes. The JVM uses invokedynamic (introduced in Java 7) to link lambda call sites at runtime. The LambdaMetafactory generates the functional interface implementation dynamically. This means lambdas are more efficient — they don't require creating a new class file per use.

Method Handle

Syntax

JAVA 8 — Lambda Syntax Forms
// Form 1: No parameters
Runnable r = () -> System.out.println("Hello");

// Form 2: Single parameter (parens optional)
Consumer<String> print = s -> System.out.println(s);

// Form 3: Multiple parameters
Comparator<Integer> cmp = (a, b) -> a - b;

// Form 4: Block body with explicit return
BiFunction<Integer, Integer, Integer> add = (a, b) -> {
    int sum = a + b;
    return sum;
};

// Form 5: Capturing effectively-final variables
int multiplier = 3; // Must be effectively final
Function<Integer, Integer> times = n -> n * multiplier;

🟢 Easy Example — Sorting with Lambda

▼
Easy 
List<String> cities = Arrays.asList("Mumbai", "Delhi", "Bangalore");

// Before Java 8
cities.sort(new Comparator<String>() {
    public int compare(String a, String b){ return a.compareTo(b); }
});

// Java 8: Lambda
cities.sort((a, b) -> a.compareTo(b));

// Even cleaner: Method reference
cities.sort(String::compareTo);

🟡 Medium Example — Lambda as Strategy Pattern

▼
Medium 
// Strategy pattern without verbose class hierarchy
@FunctionalInterface
interface DiscountStrategy {
    double apply(double price);
}

class PriceCalculator {
    public double calculate(double price, DiscountStrategy strategy) {
        return strategy.apply(price);
    }
}

// Usage: inject behavior at call site
PriceCalculator calc = new PriceCalculator();

DiscountStrategy noDiscount = price -> price;
DiscountStrategy tenPercent = price -> price * 0.9;
DiscountStrategy flatFifty  = price -> price - 50;

calc.calculate(500, tenPercent); // → 450.0

🔴 Hard Example — Lambda Variable Capture & Closure Gotcha

▼
Hard 
// GOTCHA: Lambdas capture variables, not their values over time
List<Supplier<Integer>> suppliers = new ArrayList<>();

for (int i = 0; i < 3; i++) {
    final int copy = i; // REQUIRED: i is not effectively final
    suppliers.add(() -> copy); // OK with effectively-final copy
}
// suppliers.get(0).get() → 0, get(1).get() → 1, get(2).get() → 2

// DEEP: Lambdas don't capture 'this' the same way as anon classes
class Outer {
    int x = 10;
    
    Runnable lambdaRef() {
        // 'this' refers to Outer, not the lambda itself
        return () -> System.out.println(x); // accesses Outer.x
    }
    
    Runnable anonRef() {
        return new Runnable() {
            int x = 99; // shadows Outer.x
            public void run() { System.out.println(x); } // prints 99!
        };
    }
}
⚠️ Common Mistake
Variables captured by lambdas must be effectively final (not reassigned after first assignment). Attempting to mutate a captured variable causes a compile error. Use an array wrapper trick (int[] counter = {0};) for mutation, but prefer streams' reduce or collect instead.
🔑 Key Takeaways
  • Lambdas use invokedynamic — NOT anonymous inner classes. No extra .class files.
  • this inside a lambda refers to the enclosing class, not the lambda itself.
  • Captured variables must be effectively final — no reassignment after initial assignment.
  • Lambdas enable passing behavior as data — the foundation of the functional style.

Functional Interfaces

📌 Definition

An interface with exactly one abstract method (SAM — Single Abstract Method). The @FunctionalInterface annotation enforces this at compile time. Default and static methods don't count toward the SAM contract.

Built-in Functional Interfaces (java.util.function)

InterfaceMethodInput → OutputUse Case
Function<T,R>R apply(T t)T → RTransform data
Predicate<T>boolean test(T t)T → booleanFilter conditions
Consumer<T>void accept(T t)T → voidSide effects (print, save)
Supplier<T>T get()() → TLazy value creation
BiFunction<T,U,R>R apply(T t, U u)T,U → RTwo-input transforms
UnaryOperator<T>T apply(T t)T → T (same type)Modify in place logically
BinaryOperator<T>T apply(T t1, T t2)T,T → TReduce/combine same-type
JAVA 8 — Composing Functional Interfaces
// Function composition — critical for building pipelines
Function<String, String> trim = String::trim;
Function<String, String> upper = String::toUpperCase;
Function<String, Integer> length = String::length;

// andThen: f.andThen(g) = g(f(x))
Function<String, String> trimThenUpper = trim.andThen(upper);

// compose: f.compose(g) = f(g(x))  
Function<String, Integer> pipeline = length.compose(trimThenUpper);
pipeline.apply("  hello  "); // → 5 (trimmed "HELLO".length())

// Predicate composition
Predicate<String> notEmpty = s -> !s.isEmpty();
Predicate<String> longerThan5 = s -> s.length() > 5;

Predicate<String> valid = notEmpty.and(longerThan5);
Predicate<String> invalid = valid.negate();
✅ Best Practice
Always annotate custom functional interfaces with @FunctionalInterface. It documents intent AND causes a compile error if someone accidentally adds a second abstract method, breaking backward compatibility.
🔑 Key Takeaways
  • Prefer built-in java.util.function interfaces over creating custom ones.
  • andThen and compose allow pipeline composition without nesting.
  • Primitive variants (IntFunction, LongConsumer, etc.) avoid boxing overhead.

§ 04

Stream API

📌 Definition

A Stream is a pipeline of elements supporting sequential and parallel aggregate operations. Streams don't store data — they process it lazily. They are single-use: once consumed, a stream cannot be reused.

Stream Pipeline Architecture

📦 Source
Collection, Array, Files, Generators
→
🔄 Intermediate Ops (LAZY)
filter, map, flatMap, sorted, distinct, peek, limit, skip
→
🔄 More Intermediate Ops...
Chained, not executed yet
→
✅ Terminal Op (TRIGGERS EXECUTION)
collect, forEach, reduce, count, findFirst, anyMatch, toArray
JAVA 8 — Stream Creation Methods
// From Collection
Stream<String> s1 = list.stream();

// From Array
Stream<String> s2 = Arrays.stream(new String[]{"a", "b"});

// From values
Stream<Integer> s3 = Stream.of(1, 2, 3);

// Infinite — generated lazily
Stream<Integer> natural = Stream.iterate(0, n -> n + 1);
Stream<Double> randoms = Stream.generate(Math::random);

// Primitive streams (avoid boxing!)
IntStream range = IntStream.range(0, 10);    // 0..9
LongStream lrange = LongStream.rangeClosed(1, 5); // 1..5

Intermediate vs Terminal Operations

TypeOperationsReturnsLazy?
Intermediatefilter, map, flatMap, sorted, distinct, peek, limit, skipStream✅ Yes
Terminalcollect, forEach, reduce, count, findFirst, anyMatch, toArray, min, maxResult / void❌ Triggers all
Short-circuitfindFirst, findAny, anyMatch, allMatch, noneMatch, limitResult✅ Early exit
⚠️ Critical: Streams are Single-Use
Once a terminal operation is called, the stream is consumed and closed. Calling any method on it throws IllegalStateException: stream has already been operated upon or closed. Always re-create the stream from the source.

§ 05

Method References

📌 Definition

A shorthand notation for a lambda that calls a single existing method. Cleaner than lambdas when the lambda body is just a method call. Uses the :: operator.

TypeSyntaxLambda Equivalent
Static methodInteger::parseInts -> Integer.parseInt(s)
Instance (specific object)obj::methodx -> obj.method(x)
Instance (arbitrary object)String::toUpperCases -> s.toUpperCase()
ConstructorArrayList::new() -> new ArrayList()
JAVA 8 — Method References in Practice
// Static method reference
list.stream().map(Integer::parseInt);        // String → int

// Arbitrary instance method (first param becomes the receiver)
list.stream().map(String::toLowerCase);     // each string lowered

// Specific instance method
String prefix = "Hello-";
list.stream().map(prefix::concat);           // prepend to each

// Constructor reference — used with Supplier/Function
Supplier<List<String>> listFactory = ArrayList::new;
list.stream().collect(Collectors.toCollection(TreeSet::new));
✅ Use When
  • Lambda body is a single method call
  • Method signature exactly matches functional interface
  • Improves readability over verbose lambda
🚫 Avoid When
  • Adding any transformation around the call
  • Debugging — method refs are harder to step through
  • Team isn't familiar with :: syntax

§ 06

Default Methods in Interfaces

📌 Definition

Interface methods with a concrete implementation using the default keyword. Enables adding new methods to existing interfaces without breaking all implementing classes — critical for the Java Collections API evolution.

Why It Was Introduced

Without default methods, adding stream(), forEach(), and removeIf() to the Collection interface would have broken every library and application implementing Collection. Default methods solved the "interface evolution" problem elegantly.

JAVA 8 — Default Methods
interface Validator<T> {
    boolean validate(T value); // abstract — must implement

    // Default method — free to inheritors
    default Validator<T> and(Validator<T> other) {
        return value -> this.validate(value) && other.validate(value);
    }

    default Validator<T> or(Validator<T> other) {
        return value -> this.validate(value) || other.validate(value);
    }

    // Static factory method in interface (also new in Java 8)
    static Validator<String> notEmpty() {
        return s -> s != null && !s.isEmpty();
    }
}

// Usage: compose validators
Validator<String> emailValidator =
    Validator.notEmpty().and(s -> s.contains("@")).and(s -> s.length() < 100);

emailValidator.validate("user@example.com"); // true
⚠️ Diamond Problem
If a class implements two interfaces with the same default method, it must override and resolve the conflict explicitly. The compiler enforces this. Use InterfaceA.super.method() to delegate to a specific interface's implementation.
🔑 Key Takeaways
  • Default methods enable non-breaking interface evolution — the "open/closed" principle at the language level.
  • Class implementations always win over interface defaults (class > interface in method resolution).
  • More specific interfaces win when two defaults exist (child interface > parent interface).

§ 07

Optional Class

📌 Definition

A container object that may or may not contain a non-null value. Forces the API consumer to explicitly handle the absence case, eliminating NullPointerException at the API boundary.

The Problem: Null Is Broken

Tony Hoare called null his "billion-dollar mistake." Java's Optional is NOT a null replacement everywhere — it's a return type signal that says "this method might not return a value."

JAVA 8 — Optional Patterns
// Creation
Optional<String> present = Optional.of("hello");         // throws if null
Optional<String> nullable = Optional.ofNullable(value);  // safe
Optional<String> empty = Optional.empty();

// ❌ Anti-pattern: isPresent + get = pointless Optional
if (opt.isPresent()) { String v = opt.get(); ... }  // just use null!

// ✅ Correct patterns
opt.orElse("default");                              // value or default
opt.orElseGet(() -> computeDefault());               // lazy default
opt.orElseThrow(() -> new NotFoundException());    // or throw
opt.map(String::toUpperCase);                       // transform if present
opt.flatMap(s -> findUser(s));                      // chain Optionals
opt.filter(s -> s.length() > 5);                    // conditional keep
opt.ifPresent(s -> logger.debug(s));              // side-effect if present

// Real-world: chained lookups without NPE
Optional<String> city = findUser(userId)
    .flatMap(User::getAddress)
    .map(Address::getCity)
    .filter(c -> !c.isEmpty());
🚫 Do NOT Use Optional As
Field type (doubles serialization size), method parameter (use overloads instead), collection element (use empty collection), or in performance-critical hot paths (boxing overhead).

§ 08

Date and Time API (java.time)

📌 Definition

A comprehensive, immutable, thread-safe date/time API replacing the broken java.util.Date and Calendar. Designed by the creators of Joda-Time (JSR-310).

Why The Old API Was Broken

  • Date is mutable → thread-unsafe, defensive copies everywhere
  • Months are 0-indexed in Calendar → constant off-by-one bugs
  • No timezone-aware vs local-time separation
  • No concept of date-only or time-only values
JAVA 8 — java.time Core Classes
// LocalDate: date only, no time, no timezone
LocalDate today = LocalDate.now();
LocalDate dob = LocalDate.of(1990, Month.MARCH, 15);
Period age = Period.between(dob, today);  // years, months, days

// LocalTime: time only
LocalTime now = LocalTime.now();
LocalTime noon = LocalTime.of(12, 0);

// LocalDateTime: date + time, NO timezone
LocalDateTime ldt = LocalDateTime.now();

// ZonedDateTime: date + time + timezone (USE FOR STORAGE)
ZonedDateTime zdt = ZonedDateTime.now(ZoneId.of("Asia/Kolkata"));

// Instant: machine time, nanoseconds from epoch (UTC)
Instant ts = Instant.now();

// Date arithmetic (immutable — returns NEW instance)
LocalDate nextWeek = today.plusWeeks(1);
LocalDate lastMonth = today.minusMonths(1);
boolean isLeap = today.isLeapYear();

// DateTimeFormatter (thread-safe!)
DateTimeFormatter fmt = DateTimeFormatter.ofPattern("dd/MM/yyyy");
String formatted = today.format(fmt);
LocalDate parsed = LocalDate.parse("15/03/2024", fmt);
✅ Storage Rule
Store timestamps as Instant (UTC epoch) in databases. Convert to ZonedDateTime for display using the user's timezone. Never store LocalDateTime — it's timezone-ambiguous.

§ 09

forEach() Method

📌 Definition

A default method on Iterable (and Map) that accepts a Consumer and applies it to each element. Enables internal iteration — the collection drives traversal, not the caller.

JAVA 8 — forEach Variants
// List forEach
list.forEach(item -> System.out.println(item));
list.forEach(System.out::println); // method ref form

// Map forEach (BiConsumer)
map.forEach((key, value) -> logger.info("{} → {}", key, value));

// When forEach is NOT enough: need index → use IntStream
IntStream.range(0, list.size())
    .forEach(i -> System.out.printf("%d: %s%n", i, list.get(i)));

// NEVER: modify source collection inside forEach → ConcurrentModificationException
// USE removeIf() instead:
list.removeIf(s -> s.isEmpty()); // safe structural modification

Nashorn JavaScript Engine

📌 Definition

A high-performance JavaScript runtime built into the JDK replacing the older Rhino engine. Allows executing JavaScript from Java and vice versa. Deprecated in Java 11, removed in Java 15.

JAVA 8 — Nashorn Usage
ScriptEngine engine = new ScriptEngineManager().getEngineByName("nashorn");

// Execute JS
engine.eval("print('Hello from Nashorn')");

// Call Java from JS
engine.eval("var list = java.util.Arrays.asList(1,2,3); print(list)");

// Invoke JS function from Java
engine.eval("function greet(name){ return 'Hello, ' + name; }");
Invocable inv = (Invocable) engine;
String result = (String) inv.invokeFunction("greet", "World");
⚠️ Status: Deprecated & Removed
Nashorn is deprecated since Java 11 and removed in Java 15. For JS in Java today, use GraalVM's Polyglot API. Don't build new features on Nashorn.

StringJoiner

📌 Definition

A utility class for constructing a string with configurable delimiter, prefix, and suffix. Powers String.join() and Collectors.joining() under the hood.

JAVA 8 — StringJoiner
// Basic
StringJoiner sj = new StringJoiner(", ", "[", "]");

sj.toString(); // → "[Alice, Bob, Charlie]"

// Empty value
StringJoiner empty = new StringJoiner(",", "[", "]");
empty.setEmptyValue("N/A");
empty.toString(); // → "N/A"

// Merge two joiners
StringJoiner j1 = new StringJoiner(" | ");

j1.add("A"); j2.add("B");
j1.merge(j2).toString(); // → "A | B"

// In streams (most common usage)
String csv = names.stream().collect(Collectors.joining(", "));

§ 12

Parallel Streams

📌 Definition

A stream that uses the Fork/Join common pool to split the data source, process chunks in parallel across CPU cores, then merge results. One method call away from sequential streams.

JAVA 8 — Parallel Streams
// Sequential → Parallel (just add .parallel())
long count = list.parallelStream()
    .filter(s -> s.startsWith("A"))
    .count();

// Convert mid-pipeline
list.stream()
    .filter(...)
    .parallel()    // switch to parallel here
    .map(...)
    .sequential()  // switch back
    .collect(...);

// Controlling thread pool size (system property)
// -Djava.util.concurrent.ForkJoinPool.common.parallelism=4

// Custom pool (avoid monopolizing common pool)
ForkJoinPool customPool = new ForkJoinPool(4);
customPool.submit(() ->
    list.parallelStream().filter(...).collect(Collectors.toList())
).get();
🚫 Parallel Stream Pitfalls
Never use parallel streams with: stateful lambdas, shared mutable state, I/O operations, small datasets (<10,000 elements), ordered-sensitive operations (forEachOrdered is slower than forEach). The Fork/Join overhead can make parallel slower than sequential for small/simple tasks.
✅ Good for Parallel
  • CPU-bound, stateless operations
  • Large datasets (>100k elements)
  • Order doesn't matter
  • Independent computations per element
🚫 Bad for Parallel
  • I/O-bound operations
  • Shared mutable state
  • Small collections
  • LinkedList (poor splittability)

Collectors API

📌 Definition

Factory methods that create Collector instances — strategies for accumulating stream elements into result containers (lists, maps, strings, summaries, groups).

JAVA 8 — Collectors Comprehensive Guide
// ── Basic Collection ──────────────────────────────────────
List<String> list = stream.collect(Collectors.toList());
Set<String> set   = stream.collect(Collectors.toSet());
TreeSet<String> sorted = stream.collect(Collectors.toCollection(TreeSet::new));

// ── Joining ───────────────────────────────────────────────
String csv  = stream.collect(Collectors.joining(", "));
String wrap = stream.collect(Collectors.joining(", ", "[", "]"));

// ── Grouping (most powerful) ──────────────────────────────
Map<Department, List<Employee>> byDept =
    employees.stream().collect(Collectors.groupingBy(Employee::getDept));

// groupingBy with downstream collector
Map<Department, Long> countByDept =
    employees.stream().collect(
        Collectors.groupingBy(Employee::getDept, Collectors.counting())
    );

// Multi-level grouping
Map<Dept, Map<City, List<Emp>>> nested =
    emps.stream().collect(
        Collectors.groupingBy(Emp::getDept,
            Collectors.groupingBy(Emp::getCity))
    );

// ── Partitioning ─────────────────────────────────────────
Map<Boolean, List<Integer>> evenOdd =
    numbers.stream().collect(Collectors.partitioningBy(n -> n % 2 == 0));
// {true=[2,4,6], false=[1,3,5]}

// ── Statistics ────────────────────────────────────────────
IntSummaryStatistics stats = nums.stream()
    .collect(Collectors.summarizingInt(Integer::intValue));
stats.getAverage(); stats.getMax(); stats.getSum(); stats.getCount();

// ── toMap ─────────────────────────────────────────────────
Map<Integer, String> idToName =
    users.stream().collect(Collectors.toMap(
        User::getId,
        User::getName,
        (existing, replacement) -> existing  // merge fn for duplicates
    ));
🔑 Key Takeaways
  • groupingBy with a downstream collector is the most powerful Collectors pattern.
  • Always provide a merge function in toMap to handle potential duplicate keys.
  • Collectors.toUnmodifiableList() (Java 10+) is preferred over toList() for defensive copies.

Spliterator (Advanced)

📌 Definition

An object for traversing and partitioning elements of a source. The backbone of the Stream API — every stream uses a Spliterator internally. Spliterator = Iterator + splitting logic for parallel decomposition.

JAVA 8 — Custom Spliterator
// Custom Spliterator for a word-range in a String
class WordSpliterator implements Spliterator<String> {
    private final String text;
    private int current;

    public WordSpliterator(String text, int start) {
        this.text = text; this.current = start;
    }

    @Override
    public boolean tryAdvance(Consumer<? super String> action) {
        while (current < text.length() && text.charAt(current) == ' ') current++;
        if (current == text.length()) return false;
        int start = current;
        while (current < text.length() && text.charAt(current) != ' ') current++;
        action.accept(text.substring(start, current));
        return true;
    }

    @Override
    public Spliterator<String> trySplit() {
        int mid = (current + text.length()) / 2;
        // Find word boundary
        while (mid < text.length() && text.charAt(mid) != ' ') mid++;
        if (mid >= text.length()) return null;
        WordSpliterator split = new WordSpliterator(text.substring(0, mid), current);
        current = mid;
        return split;
    }

    @Override public long estimateSize() { return Long.MAX_VALUE; }
    @Override public int characteristics() { return ORDERED | NONNULL; }
}

// Use it with StreamSupport
Stream<String> words = StreamSupport.stream(
    new WordSpliterator("hello world java eight", 0), false);
words.forEach(System.out::println);
ℹ️ Spliterator Characteristics
Key characteristics: ORDERED, DISTINCT, SORTED, SIZED, NONNULL, IMMUTABLE, CONCURRENT, SUBSIZED. Providing accurate characteristics lets the Stream engine optimize the pipeline — e.g., SIZED enables better parallel splitting.

CompletableFuture

📌 Definition

A Future that can be explicitly completed, composed, and chained. Enables non-blocking async programming with callbacks — replacing callback-hell with composable promise-style pipelines.

Why It Was Needed

Future.get() is blocking — it defeats the purpose of async. CompletableFuture adds callback chaining (thenApply, thenCompose), error handling (exceptionally), and parallel composition (allOf, anyOf) — all non-blocking.

supplyAsync()
→
thenApply()
transform result
→
thenCompose()
chain async
→
thenAccept()
consume result
JAVA 8 — CompletableFuture Patterns
// ── Basic Async ───────────────────────────────────────────
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> fetchUserFromDB(userId))  // runs in ForkJoin pool
    .thenApply(User::getName)                    // transform (non-blocking)
    .thenApply(String::toUpperCase);

// ── thenCompose: flat map for async (avoids nested futures) ─
CompletableFuture<Order> order = CompletableFuture
    .supplyAsync(() -> getUser(id))
    .thenCompose(user -> getOrderAsync(user)); // user → CF<Order>

// ── thenCombine: combine two independent futures ──────────
CompletableFuture<Double> price = CompletableFuture.supplyAsync(() -> getUSD());
CompletableFuture<Double> rate  = CompletableFuture.supplyAsync(() -> getRate());
CompletableFuture<Double> inr   = price.thenCombine(rate, (p, r) -> p * r);

// ── allOf: wait for all (parallel HTTP calls) ─────────────
CompletableFuture<String> c1 = CompletableFuture.supplyAsync(() -> callServiceA());
CompletableFuture<String> c2 = CompletableFuture.supplyAsync(() -> callServiceB());
CompletableFuture<String> c3 = CompletableFuture.supplyAsync(() -> callServiceC());

CompletableFuture.allOf(c1, c2, c3)
    .thenRun(() -> {
        String a = c1.join(), b = c2.join(), c = c3.join();
        aggregate(a, b, c);
    });

// ── Error handling ────────────────────────────────────────
CompletableFuture<User> safe = CompletableFuture
    .supplyAsync(() -> riskyCall())
    .exceptionally(ex -> {
        logger.error("Failed: {}", ex.getMessage());
        return User.defaultUser();  // fallback value
    })
    .handle((result, ex) -> {  // handle both success and failure
        if (ex != null) return User.defaultUser();
        return result;
    });

// ── Custom Executor (critical for production!) ────────────
ExecutorService executor = Executors.newFixedThreadPool(10);
CompletableFuture.supplyAsync(() -> dbQuery(), executor)
    .thenApplyAsync(User::process, executor);
🚫 Production Warning: Always Use Custom Executor
By default, supplyAsync uses the common ForkJoin pool (shared with parallel streams). For I/O operations, always provide a custom executor to avoid starving the common pool and deadlocking other parts of your application.
🔑 Key Takeaways
  • thenApply = synchronous transform; thenApplyAsync = async transform on executor.
  • thenCompose = flatMap for futures — prevents CompletableFuture<CompletableFuture<T>>.
  • exceptionally handles only errors; handle handles both success and error paths.
  • Always provide custom ExecutorService for I/O-bound async tasks.

Stream API — Master Deep Dive

🧠 Internal Architecture
Streams are built on top of Spliterators (data source) + a pipeline of operations stored as a linked list of StatelessOp / StatefulOp stages. Operations are fused where possible (loop fusion). Stateful ops (sorted, distinct) break fusion boundaries.

Lazy Evaluation — The Core Principle

    .map(name -> { System.out.println("map: " + name); return name.toUpperCase(); });

// Nothing printed yet!

// SHORT-CIRCUIT: findFirst stops after first match

    .filter(name -> { System.out.println("checking: " + name); return name.startsWith("C"); })
    .findFirst();
// Prints: "checking: Alice", "checking: Bob", "checking: Charlie" then STOPS

Parallel Stream Pitfalls

JAVA 8 — Parallel Stream Anti-patterns

numbers.parallelStream().forEach(shared::add); // DATA RACE!

// ✅ CORRECT: use thread-safe collector
List<Integer> safe = numbers.parallelStream()
    .collect(Collectors.toList()); // thread-safe by design

numbers.parallelStream().forEach(n -> {
    restTemplate.getForObject("http://api...", String.class); // BLOCKS pool!

    .map(CompletableFuture::join)

Streams create intermediate objects (spliterators, pipeline nodes, closures). For hot paths (>10M calls/sec), measure with JMH. Prefer imperative loops for tight loops in low-latency systems. The overhead is negligible in most business applications.

§ 19

Interview Preparation — 30+ Q&A

Q1What is the difference between a lambda expression and an anonymous inner class?
Lambdas use invokedynamic and don't create a separate .class file. Anonymous inner classes create a new class file. Lambdas don't have their own this (refers to enclosing class); anonymous classes have their own this. Lambdas can only implement functional interfaces; anonymous classes can implement any interface or extend a class. Lambdas are more memory-efficient (no per-instance overhead from anon class).
Q2What is an effectively final variable and why is it required for lambda capture?
A variable is effectively final if it's never reassigned after initialization (even without the final keyword). Required because lambdas are closures that capture the variable value at creation time. If the variable could change, it would create a concurrency hazard — the lambda might execute in a different thread after the variable's scope has ended. The JVM needs to guarantee the captured value won't change.
Q3Explain the difference between map() and flatMap() in streams.
map(): 1-to-1 transform. Each element produces exactly one output element. Stream<T> → Stream<R>. flatMap(): 1-to-many transform followed by flattening. Each element produces a Stream, all streams are concatenated. Stream<T> → Stream<Stream<R>> → Stream<R>. Use flatMap when you'd otherwise get nested collections (e.g., each Order has multiple Items — you want a flat Stream<Item>, not Stream<List<Item>>).
Q4When would you NOT use parallel streams?
Avoid parallel streams when: (1) Dataset is small (<10k elements — overhead outweighs benefit); (2) Order matters and operations aren't commutative; (3) Operations involve I/O (blocks the shared ForkJoin pool); (4) Operations mutate shared state (race conditions); (5) Source has poor splittability (LinkedList); (6) Operations are cheap (thread coordination overhead dominates); (7) You're in a server container (Tomcat already manages threads — stealing from common pool is dangerous).
Q5What is the difference between thenApply(), thenCompose(), and thenCombine() in CompletableFuture?
thenApply(fn): Like map — transforms the result synchronously when done. Returns CF<R>. thenCompose(fn): Like flatMap — fn returns a CF<R>, and it flattens to avoid CF<CF<R>>. Use for chaining async operations. thenCombine(cf2, fn): Combines results of TWO independent futures when both complete. Use when you have parallel independent async tasks whose results need to be merged.
Q6What is the contract of a Functional Interface? Can it have default methods?
A functional interface must have exactly ONE abstract method (SAM). It CAN have any number of default and static methods. It CAN override methods from Object (equals, hashCode, toString) without counting as abstract methods. @FunctionalInterface is optional but recommended — it enforces the contract at compile time and documents intent.
Q7How does Optional.orElse() differ from Optional.orElseGet()?
orElse(value): The default value is evaluated EAGERLY — computed before Optional is checked. Even if the Optional has a value, the default expression runs. orElseGet(supplier): The supplier is evaluated LAZILY — only called when the Optional is empty. Always prefer orElseGet when the default involves computation, DB call, or object creation, to avoid unnecessary work.
Q8What are the characteristics of a Spliterator and why do they matter?
Characteristics are bitwise flags that describe the data source: ORDERED, DISTINCT, SORTED, SIZED, NONNULL, IMMUTABLE, CONCURRENT, SUBSIZED. They matter because the Stream engine uses them to optimize the pipeline — e.g., knowing DISTINCT skips adding a HashSet in distinct(); SIZED enables better parallel partitioning; SORTED can skip re-sorting. Providing inaccurate characteristics causes incorrect results.
Q9Explain the diamond problem with default methods and how Java resolves it.
If class C implements interfaces A and B, both with the same default method, Java forces C to override the method (compile error if not). Resolution rules: (1) Class wins over interface; (2) Most specific interface wins (child interface beats parent interface); (3) If still ambiguous, explicit override required using A.super.method() or B.super.method() syntax.
Q10What is method reference and what are its 4 types?
(1) Static: ClassName::staticMethod — equivalent to x -> ClassName.staticMethod(x). (2) Instance (specific object): instance::method — equivalent to x -> instance.method(x). (3) Instance (arbitrary object): ClassName::instanceMethod — equivalent to (x, y) -> x.instanceMethod(y); first param becomes receiver. (4) Constructor: ClassName::new — equivalent to () -> new ClassName().
Q11What is the difference between Collectors.toList() and Collectors.toUnmodifiableList()?
toList() returns a mutable List — you can add/remove elements. toUnmodifiableList() (Java 10+) returns an unmodifiable view — any structural modification throws UnsupportedOperationException. Both allow null elements. Note: In Java 16+, Stream.toList() directly returns an unmodifiable, null-forbidding list.
Q12How does Stream.reduce() work? What are the identity and combiner arguments?
reduce(identity, accumulator): Identity is the initial value (must be neutral for the operation — 0 for sum, 1 for product, empty string for concat). Accumulator combines identity with each element. reduce(identity, accumulator, combiner): The combiner merges partial results in parallel processing — it combines results from different threads. The combiner must be compatible with the accumulator.
Q13Scenario: You have a List<CompletableFuture<String>>. How do you wait for all and collect results?
List<CompletableFuture<String>> futures = ...;
CompletableFuture<Void> allDone = CompletableFuture.allOf(
    futures.toArray(new CompletableFuture[0]));
List<String> results = allDone.thenApply(v -> 
    futures.stream().map(CompletableFuture::join)
        .collect(Collectors.toList())).join();
Use join() instead of get() — it throws unchecked exceptions.
Q14What is the difference between LocalDate, LocalDateTime, and ZonedDateTime? Which should you store in a DB?
LocalDate: Date only, no time, no timezone (birthdays, deadlines). LocalDateTime: Date + time, no timezone (schedule in abstract time). ZonedDateTime: Full date + time + timezone — represents a specific instant on Earth. Store as Instant (epoch milliseconds/UTC) in DB — it's unambiguous. Convert to ZonedDateTime using user's timezone for display only.
Q15What is lazy evaluation in streams and why is it important?
Intermediate operations (filter, map, etc.) don't execute until a terminal operation is called. The stream engine builds a pipeline graph, then when terminal runs, elements flow through ALL operations one at a time (loop fusion). Importance: (1) Short-circuit optimization — findFirst() stops processing after first match; (2) No intermediate collections created; (3) Infinite streams become possible (generate/iterate with limit). Without laziness, you'd need to materialize every intermediate step.
Q16What happens if you call a terminal operation twice on the same stream?
IllegalStateException: stream has already been operated upon or closed. Streams are single-use. Once a terminal operation is invoked, the stream is consumed and cannot be reused. Always re-create from the source. If you need to process the same data multiple times, materialize to a collection first, or use Supplier<Stream<T>> pattern.
Q17What is Collectors.groupingBy()? Write an example grouping employees by department, counting each.
groupingBy partitions elements into a Map based on a classifier function. With a downstream collector it further processes each group.
Map<String, Long> countByDept = employees.stream()
    .collect(Collectors.groupingBy(
        Employee::getDept,
        Collectors.counting()));
Q18What is Function.andThen() vs Function.compose()?
f.andThen(g) = apply f first, then g on result: g(f(x)). f.compose(g) = apply g first, then f on result: f(g(x)). Memory trick: andThen = "after this, do that" (natural reading order). compose = mathematical composition (right to left).
Q19How is CompletableFuture.exceptionally() different from handle()?
exceptionally(fn): Called ONLY when the future completes exceptionally. Provides a fallback value. Doesn't see the successful result. handle(BiFunction<T,Throwable,R>): Called in BOTH cases — success and failure. First param is result (null on failure), second is exception (null on success). Prefer handle when you need to transform both paths.
Q20Scenario: Stream processing throws an exception mid-pipeline. How do you handle it?
Streams don't have built-in checked exception handling. Options: (1) Wrap in unchecked exception inside lambda; (2) Use a try-catch inside the lambda to return Optional/null and filter downstream; (3) Create a wrapper utility: static <T,R> Function<T,R> wrap(CheckedFunction<T,R> fn) that catches checked and wraps in RuntimeException; (4) Use Vavr's Try monad for functional error handling without exceptions.
Q21What is the difference between Predicate.and() and &&? When would they behave differently?
Both short-circuit (right side not evaluated if left is false). Key difference: Predicate.and() allows building composable, reusable predicates that can be passed around as values. && is an inline boolean expression. Behavioral difference: Predicate.and() can be composed at runtime from different sources; && is fixed at compile time. Use and() for validator chains, filter pipelines, and configuration-driven filtering.
Q22Why should you avoid Optional as a method parameter?
Optional as parameter forces callers to wrap values in Optional (boilerplate) but doesn't prevent passing Optional.empty() OR null (the original problem). Method overloading is cleaner: process(User user) + process(). Optional was designed as a return type to signal "might not have a value." As a parameter it's leaking implementation detail into the API contract and adds no real safety benefit.
Q23What is the purpose of Collectors.collectingAndThen()?
Applies a finisher function to the result of another collector. Common use: make the collected result unmodifiable. Example: Collectors.collectingAndThen(Collectors.toList(), Collections::unmodifiableList). Also useful for converting the collected result to another type, or applying post-processing to grouped results.
Q24How do you create a custom Collector?
Implement Collector<T, A, R> interface with: supplier() — creates mutable result container; accumulator() — adds element to container; combiner() — merges two containers (for parallel); finisher() — converts container to final result; characteristics() — hints (CONCURRENT, IDENTITY_FINISH, UNORDERED). Or use the factory: Collector.of(supplier, accumulator, combiner, finisher, characteristics).
Q25Scenario: Find the second-highest salary using streams.
Optional<Integer> secondHighest = employees.stream()
    .map(Employee::getSalary)
    .distinct()  // remove duplicates
    .sorted(Comparator.reverseOrder())
    .skip(1)     // skip highest
    .findFirst();
Important: use distinct() before sorting if duplicates of the max salary exist and you want the truly second-unique value. Skip without distinct may skip a duplicate of the max.
Q26What is peek() and when should you use it?
peek() is an intermediate operation that applies a Consumer (side effect) without transforming the element. Primary use: debugging — print elements at various pipeline stages. Warning: peek() relies on the pipeline actually being evaluated (lazy). Don't use peek() for logic — only for debugging/logging. In production, prefer structured logging via map + log + map back.
Q27How does the new Date/Time API handle thread safety vs the old API?
java.util.Date and Calendar are mutable and NOT thread-safe — require synchronization or ThreadLocal for each use. SimpleDateFormat is notoriously thread-unsafe. Java 8's java.time classes are all immutable and thread-safe by design. DateTimeFormatter instances are thread-safe and can be stored as static constants. No more defensive copying or ThreadLocal wrapping needed.
Q28What is the difference between anyMatch, allMatch, and noneMatch?
All are short-circuiting terminal operations returning boolean. anyMatch(predicate): true if ANY element matches — stops at first true. allMatch(predicate): true if ALL elements match — stops at first false. noneMatch(predicate): true if NO element matches — stops at first true. On empty stream: anyMatch → false, allMatch → true, noneMatch → true (vacuous truth).
Q29Scenario: A service returns CF<User>, and User has a method getOrders() returning CF<List<Order>>. How do you chain them?
Use thenCompose — it flat-maps nested CompletableFutures:
CompletableFuture<List<Order>> orders = 
    userService.getUser(id)
        .thenCompose(User::getOrders);
Using thenApply would give CF<CF<List<Order>>> — a nested future that requires two .join() calls. thenCompose is the async equivalent of flatMap.
Q30How has Java 8 influenced Java 17 and Java 21?
Java 8 set the functional foundation: Records (Java 16) use lambdas internally. Sealed classes (Java 17) pair with pattern matching. Stream API got new methods: takeWhile, dropWhile (Java 9), iterate with predicate (Java 9). Optional got ifPresentOrElse, stream(), or() (Java 9). Virtual threads (Java 21) work seamlessly with CompletableFuture patterns. The functional style normalized by Java 8 made all of these additions feel natural.
Q31Scenario-based: Your team wants to use parallelStream() on everything for "performance." How do you respond?
Push back with data. Run a JMH benchmark. Parallel streams have coordination overhead — thread splitting, result merging, ForkJoin pool management. For small collections or cheap operations, parallel is SLOWER. Critical risks: (1) Common ForkJoin pool is shared — blocking operations starve other parts of the app; (2) Stateful lambdas cause race conditions; (3) Server containers (Tomcat, Jetty) already use thread pools — parallel streams are redundant or harmful. Guideline: only use parallel for CPU-bound operations on >10k elements where order doesn't matter, and measure first.

Common Mistakes

🚫 Overusing Streams
// ❌ Stream for trivial single-element check (use contains!)
boolean found = list.stream().anyMatch(x -> x.equals("target")); // OK but...
list.contains("target"); // ✅ simpler, O(1) for Set

// ❌ Collecting to list just to iterate (use forEach directly)
list.stream().filter(...).collect(Collectors.toList()).forEach(...);
list.stream().filter(...).forEach(...); // ✅ no intermediate collection

// ❌ Stream inside stream (N×M complexity) — use join/map instead
outerList.stream().map(o ->
    innerList.stream().filter(i -> i.getId().equals(o.getId())).findFirst()
); // O(n*m) — build a Map first!
🚫 Misusing Optional
// ❌ Using Optional.get() without check → NoSuchElementException
String val = optional.get(); // DANGEROUS

// ❌ Optional as field / parameter / collection element
private Optional<String> name; // DON'T — doubles serialization size
void process(Optional<User> user) {} // DON'T — use overloads

// ❌ isPresent + get = defeating the purpose
if (opt.isPresent()) { return opt.get(); } // just use null checks then

// ✅ Correct usage
return opt.orElseThrow(() -> new NotFoundException("not found"));
return opt.map(User::getName).orElse("Anonymous");
🚫 Parallel Stream Issues
// ❌ Shared mutable accumulator in parallel
List<String> result = new ArrayList<>();
stream.parallel().forEach(result::add); // RACE CONDITION

// ❌ Stateful lambda (depends on external state)
int[] count = {0};
stream.parallel().forEach(e -> count[0]++); // NOT thread-safe

// ❌ I/O in parallel stream (blocks common pool)
users.parallelStream().forEach(u -> repo.save(u)); // DANGER

// ✅ Use Collectors for thread-safe aggregation
List<String> safe = stream.parallel().collect(Collectors.toList());
long count = stream.parallel().filter(...).count(); // thread-safe
🚫 Stream Reuse
// ❌ Reusing a stream after terminal op → IllegalStateException
Stream<String> stream = list.stream().filter(String::isEmpty);
stream.count();     // terminal op → stream consumed
stream.forEach(...); // THROWS: stream has already been operated upon

// ✅ Use Supplier to recreate
Supplier<Stream<String>> streamSup = () -> list.stream().filter(String::isEmpty);
streamSup.get().count();
streamSup.get().forEach(...);

Best Practices — Senior Level

🎯

Favor primitives

Use IntStream, LongStream, DoubleStream to avoid boxing overhead in number-heavy operations.

🔒

Immutability first

Prefer toUnmodifiableList() and immutable data in streams. Never mutate shared state inside lambdas.

📏

Keep lambdas short

If a lambda exceeds 3 lines, extract to a named private method. Enhances readability and testability.

⚡

Parallel: measure first

Never add parallelStream() without JMH benchmarks. Overhead is real — only CPU-bound + large data.

🏭

Custom executor

For I/O-bound CompletableFuture tasks, always provide a dedicated ExecutorService, never the common pool.

📦

Optional as return only

Use Optional only as a return type signal. Never for fields, parameters, or collection elements.

🗂

Prefer method refs

Use method references when the lambda is just a single method call — cleaner and the JVM may optimize better.

🧪

Test with JMH

Benchmark streams vs loops for hot paths. JVM JIT makes micro-benchmarks tricky — always use proper harness.

When to Use Streams vs Loops (Decision Guide)

✅ Use Streams When
  • Transforming, filtering, grouping collections
  • Multiple chained operations on data
  • Need parallel processing (CPU-bound, large data)
  • Expressing WHAT not HOW (declarative style)
  • Working with infinite/lazy sequences
🚫 Use Loops When
  • Need to break/continue mid-iteration
  • Mutating the source collection
  • Exception handling without workarounds
  • Need index access (use IntStream.range instead)
  • Extremely performance-sensitive hot path
JAVA 8 — Anti-patterns to Avoid
// ❌ ANTI-PATTERN: Stream inside stream (O(n²))
orders.stream()
    .filter(o -> items.stream().anyMatch(i -> i.getOrderId().equals(o.getId())))
    .collect(...);
// ✅ FIX: Build a Set first
Set<String> itemOrderIds = items.stream().map(Item::getOrderId).collect(Collectors.toSet());
orders.stream().filter(o -> itemOrderIds.contains(o.getId())).collect(...);

// ❌ ANTI-PATTERN: Collecting then streaming again
users.stream().filter(...).collect(Collectors.toList()).stream().map(...).collect(...);
// ✅ FIX: Chain in one pipeline
users.stream().filter(...).map(...).collect(...);

// ❌ ANTI-PATTERN: Using streams for simple existence check
boolean exists = users.stream().filter(u -> u.getId().equals(id)).count() > 0;
// ✅ FIX: Short-circuit
boolean exists = users.stream().anyMatch(u -> u.getId().equals(id));

Final Summary

🔥 Why Java 8 Changed Everything

Java 8 was not a feature release — it was a paradigm expansion. It didn't abandon OOP; it added functional programming as a peer paradigm. The result: the same language can express both data modeling (OOP) and data processing (FP) idiomatically.

Three inventions drove this: Lambdas (functions as values), Streams (lazy, composable pipelines), and CompletableFuture (composable async). Together they solved Java's verbosity, concurrency, and expressiveness problems simultaneously.

The Ripple Effect on Modern Java

☕

Java 9–11

Stream enhancements (takeWhile, dropWhile), Optional.stream(), List.of(), var keyword, HTTP/2 client built on CF.

📦

Java 14–17

Records use lambdas internally. Sealed classes + pattern matching enable functional-style exhaustive dispatch.

🚀

Java 21

Virtual Threads (Project Loom) make every CF pattern infinitely scalable. Pattern matching completes the FP story.

🏗

Ecosystem

Spring WebFlux, Reactor, RxJava — all built on Java 8's functional interfaces and CompletableFuture model.

🎓 The Senior Engineer Mindset on Java 8
  • Lambdas are not syntax sugar — they're a new execution model via invokedynamic.
  • Streams are declarative specifications, not loops — think pipelines, not iterations.
  • Optional is a return-type contract, not a null replacement — use it at API boundaries only.
  • Parallel streams are a footgun — measure before using, always with custom executors for I/O.
  • CompletableFuture is your async DSL — compose, combine, and handle errors functionally.
  • java.time is the only correct date API — Instant for storage, ZonedDateTime for display.
  • Java 8's functional foundation underpins every modern Java feature from Java 9 through 21.
